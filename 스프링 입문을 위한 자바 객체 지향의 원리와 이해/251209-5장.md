# 5장 객체 지향 설계 5원칙 - SOLID

응집도는 높이고, 결합도는 낮추는 원칙을 객체 지향의 관점에서 재정립한 것

## SRP (Single Responsibility Principle) - 단일 책임 원칙

어떤 클래스를 변경해야 하는 이유는 오직 하나뿐이어야 한다
- SRP는 객체 지향 4대 특성 중 추상화와 가장 관계가 깊다

## OCP (Open Closed Principle) - 개방 폐쇄 원칙

소프트웨어 엔티티(클래스, 모듈, 함수 등)는 확장에 대해서는 열려 있어야 하지만, 변경에 대해서는 닫혀 있어야 한다

- 예를 들어 JDBC를 사용하는 클라이언트는 데이터베이스가 오라클에서 MySQL로 바뀌더라도 Connection을 설정하는 부분 왜에는 코드를 수정할 필요가 없다
- 자바 애플리케이션은 JDBC 인터페이스 덕분에 데이터베이스라는 주변의 변화에 닫혀 있는 것이다
- 자바에도 적용이 되어있는데 개발자는 소스코드가 어떤 운영체제 상에서 구동될지에 대해서는 걱정할 필요가 없는 것이 그 경우이다

## LSP (Liskov Substitution Principle) - 리스코프 치환 원칙

서브 타입은 언제나 자신의 기반 타입으로 교체할 수 있어야 한다

- 상속이 조직도나 계층도 형태로 구축된 경우 이를 잘 어긴다
- 정의를 의역하면 -> 하위 클래스의 인스턴스는 상위형 객체 참조 변수에 대입해 상위 클래스의 인스턴스 역할을 하는 데 문제가 없어야 한다
- 예시)
    - 안좋은 예시)
        - `아버지 춘향이 = new 딸()`
        - 딸에게 아빠의 역할을 시킬 수 있는가
    - 좋은 예시)
        - `동물 뽀로로 = new 펭귄()`
        - 펭귄은 동물의 행위를 할 수 있다

## ISP (Interface Segregation Principle) - 인터페이스 분리 원칙

클라이언트는 자신이 사용하지 않는 메소드에 의존 관계를 맺으면 안된다

- SRP와 ISP는 같은 문제에 대한 두 가지 다른 해결책으로 볼 수 있다
- 특별한 경우가 아니라면 SRP가 더 좋은 해결책이다
- ISP를 얘기할 때 항상 함께 등장하는 것이 인터페이스 최소주의 원칙이다
    - 이는 인터페이스를 통해 메소드를 외부에 제공할 때는 최소한의 메소드만 제공하라는 것이다

## DIP (Dependency Inversion Principle) - 의존 역전 원칙

고차원 모듈은 저차원 모듈에 의존하면 안된다. 이 두 모듈 모두 다른 추상화된 것에 의존해야 한다
<br>
추상화된 것은 구체적인 것에 의존하면 안된다. 구체적인 것이 추상화된 것에 의존해야 한다
<br>
자주 변경되는 구체 클래스의 의존하지 마라

- 변하기 쉬운 것에 의존하던 것을 추상화된 인터페이스나 상위 클래스를 두어 변하기 쉬운 것의 변화에 영향받지 않게 하는 것
- 정의를 의역하면 -> 자신보다 변하기 쉬운 것에 의존하지 마라
- 상위 클래스일수록, 인테피읏일수록, 추상 클래스일수록 변하지 않을 가능성이 높기에 하위 클래스나 구체 클래스가 아닌 상위 클래스, 인터페이스, 추상 클래스를 통해 의존하라는 것이 의존 역전 원칙이다

## SOLID 정리

- SOLID를 얘기할 때 빼놓을 수 없는것이 SoC(Seperation of Concerns)이다
- 관심이 같은 것끼리는 하나의 객체 안으로 또는  친한 객체로 모으고, 관심이 다른 것은 가능한 한 따로 떨어져 서로 영향을 주지 않도록 분리하라는 것
- 하나의 속성, 하나의 메소드, 하나의 크래스, 하나의 모듈, 또는 하나의 패키지에는 하나의 관심사만 들어 있어야 한다는 것
