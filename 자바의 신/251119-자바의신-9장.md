# 패키지

## 패키징을 하는 이유

패키징을 하는 이유는 단순히 폴더 나누기가 아닌 대규모 코드 관리, 협업, 배포, 보안 측면에서 필수적인 구조를 만들기 위해서다

1. 클래스 이름 충돌 방지 (네임스페이스 역할)
2. 프로젝트 구조 체계화 (폴더 구조 = 패키지 구조)
3. 접근 제어(캡슐화)를 위한 단위
4. 재사용성과 모듈성 향상
    - 코드를 모듈(기능 단위)별로 분리하면 라이브러리로 만들 때, 배포할 때 유용
5. 빌드, 배포 시 패키지를 기준으로 구성된다
6. 충돌 방지 + 유지보수 + 배포까지 모든 부분에 핵심
    - 이름 충돌 방지
    - 코드 구조화
    - 접근 제어 관리
    - 모듈화/재사용성
    - 빌드(LIB/JAR/WAR) 구조
    - JVM 클래스 로딩 규칙 유지

## import 유의사항

- `import c.*` 이렇게 하면 `package c`로 선언한 클래스만 import 되고, 그 하위 패키지에 선언된 클래스는 import 하지 않는다

## import static

- static한 변수와 static 메소드를 사용하고자 할 때 용이하다
- 기존에는 `SubStatic.subStaticMethod()` 처럼 사용해야 하지만, import static을 사용하면 `subStaticMethod()`로 사용할 수 있다
- 만약 Package 클래스에 import한 동일한 이름의 static 변수나 static 메소드가 자신의 클래스에 있다면, 자신의 것에 있는것이 우선이다

## 클래스 접근 제어자 선언 시 유의할 점

- `Profile.java`라는 소스코드에 `Profile` 클래스 외에 다른 클래스가 public으로 선언되어 있으면 안된다
    - public 클래스는 반드시 파일 이름과 동일해야 한다
    - JVM의 클래스 로딩 규칙과 이름 충돌 방지
        - JVM은 클래스 이름 = 파일 이름 기준으로 클래스를 찾는다
    - 컴파일러가 파일 단위로 클래스 심볼을 관리한다
    - 가독성, 유지보수성의 핵심 규칙
