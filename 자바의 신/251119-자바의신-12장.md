# 모든 클래스의 부모 클래스인 Object 클래스

## toString()

- Object 클래스에 구현되어 있는 toString 메소드는 다음과 같다

```java
getClass().getName() + '@' + Integer.toHexString(hashCode())
```

- `getClass().getName()` : 현재 클래스의 패키지 이름과 클래스 이름이 나옴
- Integer.toHexString(hashCode()) : 해시코드에서 나온 int 값을 16진수로 변환함

## equals()와 ==의 차이

- ==
    - 객체의 참조(주소)가 같은지 비교
- equals()
    - 객체의 논리적 동등성을 비교하도록 설계된 메서드
    - 기본 구현은 ==과 동일하지만, 보통 값 기준으로 다시 오버라이드해서 사용한다

## equals()의 규약

equals()는 수학적인 동치 관계여야 해서 반드시 아래를 만족해야 한다

1. 반사성
    - `x.equals(x)`는 true
2. 대칭성
    - `x.equals(y)`가 true면 `y.equals(x)`도 true
3. 추이성
    - `x.equals(y)`, `y.equals(z)`가 true면 `x.equals(z)`도 true
4. 일관성
    - 값이 변하지 않으면 결과도 변하면 안됨
5. null 비교 안전성
    - `x.equals(null)`은 항상 false

## hashCode()가 필요한 이유

- HashSet / HashMap 같은 해시 기반 컬렉션은 객체의 hashCode()로 버킷(칸)을 먼저 찾고, 그 버킷 안에서 equals()로 진짜 같은 객체인지 확인한다
- 즉, hashCode()는 이 객체가 몇 번 칸에 들어갈지를 결정하는 번호를 구하는 메서드이다

## hashCode()의 규약

1. 같은 객체는 항상 같은 hashCode()를 반환해야 한다
2. **equals()가 true인 두 객체는 반드시 같은 hashCode를 가져야 한다**
3. equals가 false인 객체들은 가능한 hashCode도 다르면 좋다 (충돌 방지)

## equals/hashCode 절대 규칙

- equals()에서 사용한 필드들은 hashCode()에서도 반드시 사용해야 한다
- 예) equals()에서 name, age를 비교한다면, hashCode()도 name, age로 만들어야 한다
